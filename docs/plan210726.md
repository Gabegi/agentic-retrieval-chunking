# Indexing pipeline: extraction-skip, embedding dedup, persistent archive

Plan agreed 2026-07-21. Covers `ExtractionService` and the rest of the
extraction/chunking/embedding pipeline in `src/agentic-rag-app`.

## Goals

- Stop paying for Document Intelligence extraction on documents that are
  already indexed and unchanged.
- Stop paying for embedding calls on chunks that are already indexed and
  unchanged, even when they belong to a document that *did* change.
- Keep a persistent, full-content record of what each pipeline step
  (extraction, chunking, embedding) actually produced, per run.
- Make it possible to rebuild the Search index from blob alone, without
  re-running extraction or embedding (deferred â€” see Later notes).

## Current state (as of this plan)

- `ExtractionService.ExtractAsync` (`ExtractionService.cs`) already diffs
  extracted docs against `IndexDocumentService.GetCurrentIndexedDocumentDatesAsync`,
  but only *after* `IExtractionOrchestrator.ExtractDocumentsAsync` has already
  run â€” so the diff only saves chunking/embedding cost, not the paid DI calls
  in `PdfExtractionOrchestrator.ExtractPdfsFromBlobAsync`.
- No chunk-level content hash exists. The only hash in the codebase
  (`PdfExtractionOrchestrator.ComputeContentHash`) is a whole-PDF SHA-256,
  logged at Debug level only, compared against nothing.
- `content_vector` in the Search index schema (`IndexService.cs`) is
  `IsStored = false` â€” not retrievable from Search once written. Azure AI
  Search doesn't allow changing that attribute on an existing field without
  rebuilding the index, so vector reuse must live outside the index.
- `telemetry-reports` (dev-only, `IRunReportWriter.IsEnabled = env.IsDevelopment()`)
  holds diff/stats summaries only â€” source IDs and counts, never full
  document/chunk content. `indexing-pipeline` holds full content but is
  transient: `extracted.json`/`chunks.json` are deleted by the next activity
  right after it consumes them.
- `PdfExtractionOrchestrator`'s magnitude-check gate can currently abort a
  production run (`throw` in `ExtractDocumentsAsync`) if the processed record
  count swings too far from a saved baseline.

## Stage 1 â€” Extraction-skip

Build first, verify, then move on to later stages.

- `IExtractionOrchestrator`: add `ListSourceDocumentsAsync(ct)` â†’ cheap
  `{sourceId: LastModified}` from blob listing only, no download/OCR.
- `PdfExtractionOrchestrator`: implement it by reusing the existing
  blob-listing loop minus the download/extract call.
- Mirror the interface addition in `CsvIndexing`'s orchestrator too (dormant,
  kept structurally in sync with the active PDF path).
- `ExtractionService.ExtractAsync`: call `ListSourceDocumentsAsync` +
  `GetCurrentIndexedDocumentDatesAsync` *before* extraction, run the
  new/updated/skip/removed diff on IDs+dates alone (existing
  `CompareNewDocsNCurrentIndex` logic moves here), then pass the resulting
  "process this subset" id set into `ExtractDocumentsAsync`.
- `PdfExtractionOrchestrator.ExtractDocumentsAsync` /
  `ExtractPdfsFromBlobAsync`: accept that subset, skip download+DI call for
  anything not in it.
- Remove the magnitude-check `throw` (`PdfExtractionOrchestrator.cs:122-133`)
  â€” keep computing/logging `MagnitudeWarnings`/`RedFlags` into the report,
  never abort a run over it.
- "Removed document" detection becomes naturally correct as a side effect â€”
  driven off the full cheap listing, not off what actually got extracted.

## Stage 2 â€” Persistent archive (`indexing-artifacts` container)

- New container, own enable flag (not `IsDevelopment()`), on in both dev and
  prod â€” both environments process the real Cordaan corpus.
- Per run (`{instanceId}`), one report per step:
  - Extraction: full `ExtractionDocument`s + stats.
  - Chunking: full `ProtocolDocument`s pre-vector + stats.
  - Embedding: metadata only â€” id, content_hash, dims, truncated/retried
    flags. No raw vector floats (those live once in the vector cache, not
    duplicated here).
- Retention: keep forever for now. Revisit only if growth actually becomes a
  problem â€” estimated cost at plausible corpus scale (~2,000 source PDFs) is
  low single-digit GB even after a year of daily runs, i.e. cents to a few
  dollars a month at Hot-tier blob pricing. The real risk isn't $/GB, it's
  unbounded growth with no cleanup story â€” accepted for the step-report
  archives, but not for the vector cache (see Stage 4).

## Stage 3 â€” Hash-based embedding dedup + vector cache

Only pays off for the *updated-document* case: `CompareNewDocsNCurrentIndex`
currently reprocesses every chunk of an updated document, including pages
that didn't change. This lets untouched chunks reuse their existing vector
instead of re-embedding.

- `ProtocolDocument`: add `ContentHash` = SHA-256 of `EmbeddingText`
  (Content+Summary â€” the actual embedding input, so a hash match provably
  means an identical embedding), computed in `ChunkingService.ChunkDocuments`.
- Vector cache: `indexing-artifacts/vector-cache/{hash}.json`, **one blob per
  unique hash** â€” not one shared file. Avoids read-modify-write races and
  makes later eviction a simple targeted delete.
- Cache writes parallelized with bounded concurrency (same pattern as
  `MaxExtractionParallelism` in `PdfExtractionOrchestrator`) to keep this off
  the run's critical path and as optimized as possible.
- Before calling the embedding API: check the cache by `ContentHash` first â€”
  hit â‡’ reuse the vector, skip the call; miss â‡’ embed normally, then write
  the result to the cache.
- No Search index schema change needed â€” this deliberately avoids the
  `IsStored` immutability problem entirely.

## Stage 4 â€” Rolling full-corpus snapshot, then vector-cache eviction

- `indexing-artifacts/snapshots/{instanceId}/full-index.json` â€” the previous
  snapshot merged with this run's diff (new/updated in, removed out).
  Written every successful run, pruned to the **3 most recent** â€” an explicit
  exception to "keep forever," scoped only to this path.
- Immediately after this lands: **vector-cache eviction**. List blobs under
  `vector-cache/`, diff against the snapshot's referenced hash set, delete
  anything orphaned (i.e. its chunk no longer exists in any current
  document). Cheap once the snapshot exists, since it reuses the snapshot's
  hash list â€” not worth building standalone before Stage 4 exists, since that
  would need a separate full-corpus hash enumeration the index doesn't
  support today.

## Later / deferred (not building now, tracked for later)

- **Index rebuild from blob**: read the latest snapshot â†’ resolve each
  chunk's vector from the vector cache by hash â†’ bulk-upsert into a
  (possibly new) Search index. No re-extraction, no re-embedding needed.
  Also the path out of the `IsStored` immutability problem for any future
  schema change: create a new index, replay the snapshot into it, cut over â€”
  instead of a full re-extract-and-re-embed.
- Revisit retention/pruning on the Stage 2 step-report archives
  (extraction/chunking/embedding reports) if growth becomes an actual
  problem in practice.
