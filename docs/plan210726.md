# Indexing pipeline: extraction-skip, embedding dedup, persistent archive

Plan agreed 2026-07-21. Covers `ExtractionService` and the rest of the
extraction/chunking/embedding pipeline in `src/agentic-rag-app`.

## Goals

- Stop paying for Document Intelligence extraction on documents that are
  already indexed and unchanged.
- Stop paying for embedding calls on chunks that are already indexed and
  unchanged, even when they belong to a document that *did* change.
- Keep a persistent, full-content record of what each pipeline step
  (extraction, chunking, embedding) actually produced, per run.
- Make it possible to rebuild the Search index from blob alone, without
  re-running extraction or embedding (deferred — see Later notes).

## Current state (as of this plan)

- `ExtractionService.ExtractAsync` (`ExtractionService.cs`) already diffs
  extracted docs against `IndexDocumentService.GetCurrentIndexedDocumentDatesAsync`,
  but only *after* `IExtractionOrchestrator.ExtractDocumentsAsync` has already
  run — so the diff only saves chunking/embedding cost, not the paid DI calls
  in `PdfExtractionOrchestrator.ExtractPdfsFromBlobAsync`.
- No chunk-level content hash exists. The only hash in the codebase
  (`PdfExtractionOrchestrator.ComputeContentHash`) is a whole-PDF SHA-256,
  logged at Debug level only, compared against nothing.
- `content_vector` in the Search index schema (`IndexService.cs`) is
  `IsStored = false` — not retrievable from Search once written. Azure AI
  Search doesn't allow changing that attribute on an existing field without
  rebuilding the index, so vector reuse must live outside the index.
- `telemetry-reports` (dev-only, `IRunReportWriter.IsEnabled = env.IsDevelopment()`)
  holds diff/stats summaries only — source IDs and counts, never full
  document/chunk content. `indexing-pipeline` holds full content but is
  transient: `extracted.json`/`chunks.json` are deleted by the next activity
  right after it consumes them.
- `PdfExtractionOrchestrator`'s magnitude-check gate can currently abort a
  production run (`throw` in `ExtractDocumentsAsync`) if the processed record
  count swings too far from a saved baseline.

## Stage 1 — Extraction-skip ✅ done

- **No swappable-source design.** `agentic-rag-app`'s pipeline is PDF-only,
  permanently — CSV has its own fully separate pipeline in the standalone
  `CsvIndexing` project, and the two will never share a runtime
  implementation. `IExtractionOrchestrator` is kept only for
  `ExtractionService`'s unit-test mockability, not for swapping
  implementations — so nothing here was mirrored into `CsvIndexing`.
- `ExtractionService` owns blob access directly (`BlobContainerClient` for
  the `"documents"` container, injected via `program.cs`) and implements
  `ListDocumentsInBlobAsync` itself — cheap `{sourceId: LastModified}`
  listing, no download/OCR. This does *not* live on `IExtractionOrchestrator`
  — `PdfExtractionOrchestrator` already owns the same container for the
  expensive download+extract path, so listing stays alongside it rather than
  splitting blob access across two classes another way.
- `IExtractionOrchestrator` is down to `Source` + `ExtractDocumentsAsync(IReadOnlySet<string> sourceIdsToProcess, ct)`.
- `ExtractionService.ExtractAsync`: calls `ListDocumentsInBlobAsync` +
  `GetCurrentIndexedDocumentDatesAsync` *before* extraction, runs the
  new/updated/skip/removed diff on IDs+dates alone
  (`CompareSourceListingToIndex`), then passes the resulting "process this
  subset" id set into `ExtractDocumentsAsync`.
- `PdfExtractionOrchestrator.ExtractDocumentsAsync` /
  `ExtractPdfsFromBlobAsync`: accept that subset, skip download+DI call for
  anything not in it.
- Magnitude-shift check demoted from a gate to advisory-only:
  `PdfPipelineValidator.Passed` no longer depends on it (only
  reconciliation/error-rate can still fail a run). Since magnitude could no
  longer be overridden, the whole `overrideMagnitudeCheck` /
  `PassedExcludingMagnitude` / `magnitudeOverrideApplied` chain was dead code
  once decoupled — removed end-to-end (interfaces, `IndexingFunction`'s HTTP
  query param, `IndexRequest`/`ExtractRequest`).
- "Removed document" detection is now naturally correct as a side effect —
  driven off the full cheap listing, not off what actually got extracted (a
  doc that merely fails extraction one run is no longer mistaken for one
  withdrawn from source).

## Stage 2 — Persistent archive (`indexing-artifacts` container) ✅ done

- New container, always written unconditionally — no enable flag at all
  (dropped the originally-planned config toggle: a flag that's always true
  in every environment is dead weight, not a real switch).
- `IPipelineArtifactWriter`/`PipelineArtifactWriter`
  (`agentic-rag-app/Observability/`) — same shape as `IRunReportWriter` but
  its own container and no `IsEnabled` gate. Kept local to `agentic-rag-app`
  rather than in `IndexingShared`, matching how `RunReportWriter` itself is
  actually organized today (a local copy in the active pipeline, not the
  shared one `CsvIndexing` uses) — revisit if/when the "move clients to
  IndexingShared" refactor item lands.
- Per run (`{instanceId}`), one artifact per step, written from
  `IndexingFunction`'s three activities:
  - `{instanceId}/extraction.json` — full `ExtractionDocument`s + `ExtractionResults` stats.
  - `{instanceId}/chunking.json` — full `ProtocolDocument`s pre-vector + `ChunkingResults` stats.
  - `{instanceId}/embedding.json` — metadata only per chunk (id, document id,
    vector dimension count) + aggregate stats (truncated/retries/dim-errors).
    No raw vector floats. `content_hash` isn't in here yet — added once
    Stage 3 introduces it.
- Retention: keep forever for now. Revisit only if growth actually becomes a
  problem — estimated cost at plausible corpus scale (~2,000 source PDFs) is
  low single-digit GB even after a year of daily runs, i.e. cents to a few
  dollars a month at Hot-tier blob pricing. The real risk isn't $/GB, it's
  unbounded growth with no cleanup story — accepted for the step-report
  archives, but not for the vector cache (see Stage 4).

## Stage 3 — Hash-based embedding dedup + vector cache ✅ done

Only pays off for the *updated-document* case: `CompareSourceListingToIndex`
reprocesses every chunk of an updated document, including pages that didn't
change. This lets untouched chunks reuse their existing vector instead of
re-embedding.

- `ProtocolDocument.ContentHash` — computed property, SHA-256 hex of
  `EmbeddingText` (Content+Summary — the actual embedding input, so a hash
  match provably means an identical embedding). `[JsonIgnore]`, since the
  Search index has no `content_hash` field and uploading an unknown field
  would fail the upsert.
- `IVectorCache`/`VectorCache` (`Services/Indexing/Chunking/VectorCache.cs`,
  interface in `Services/Indexing/Interfaces/`) — one blob per unique hash
  under `indexing-artifacts/vector-cache/{hash}.json`, same container as the
  Stage 2 archive. One blob per hash (not one shared file) avoids
  read-modify-write races and makes later eviction a simple targeted delete.
- `EmbeddingService.EmbedDocumentsAsync`: splits incoming chunks into
  cache-hit (vector reused, no API call) vs. cache-miss (embedded normally,
  then written to the cache) *before* batching for the embedding API. A
  cached vector whose length no longer matches the configured embedding
  dimensions is treated as a miss, not trusted blindly (protects against a
  stale cache after a model/config change). Cache reads/writes run with
  bounded parallelism (`MaxCacheParallelism = 8`, same shape as
  `PdfExtractionOrchestrator.MaxExtractionParallelism`).
- `EmbeddingRunResult`/`EmbedUploadingResults`/`IndexRunReport` all gained a
  `CacheHits`/`VectorCacheHits` field, plus a new
  `Instrumentation.VectorCacheHits` counter — visible in the run report and
  telemetry, not just an internal optimization.
- Stage 2's `embedding.json` archive now also carries `ContentHash` per
  chunk (metadata only, no raw vector — the vector itself lives once in the
  vector cache).
- No Search index schema change needed — this deliberately avoids the
  `IsStored` immutability problem entirely.

## Stage 4 — Rolling full-corpus snapshot, then vector-cache eviction

- `indexing-artifacts/snapshots/{instanceId}/full-index.json` — the previous
  snapshot merged with this run's diff (new/updated in, removed out).
  Written every successful run, pruned to the **3 most recent** — an explicit
  exception to "keep forever," scoped only to this path.
- Immediately after this lands: **vector-cache eviction**. List blobs under
  `vector-cache/`, diff against the snapshot's referenced hash set, delete
  anything orphaned (i.e. its chunk no longer exists in any current
  document). Cheap once the snapshot exists, since it reuses the snapshot's
  hash list — not worth building standalone before Stage 4 exists, since that
  would need a separate full-corpus hash enumeration the index doesn't
  support today.

## Later / deferred (not building now, tracked for later)

- **Index rebuild from blob**: read the latest snapshot → resolve each
  chunk's vector from the vector cache by hash → bulk-upsert into a
  (possibly new) Search index. No re-extraction, no re-embedding needed.
  Also the path out of the `IsStored` immutability problem for any future
  schema change: create a new index, replay the snapshot into it, cut over —
  instead of a full re-extract-and-re-embed.
- Revisit retention/pruning on the Stage 2 step-report archives
  (extraction/chunking/embedding reports) if growth becomes an actual
  problem in practice.
